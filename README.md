# Behavioral Signal Pipeline

> **What it does:** This project is a full-stack behavioral analytics engine that captures granular user interaction signals from any webpage in real-time. The architecture follows an event-driven design pattern where a lightweight JavaScript tracker embedded on client websites sends behavioral events (page views, scroll depth, clicks, exit intent) to a NestJS REST API. The API immediately publishes these events to a RabbitMQ message queue and returns HTTP 202 Accepted, ensuring the client never blocks while waiting for processing. A background worker consumes from the queue and writes data to two storage systems in parallel: PostgreSQL maintains upserted user behavioral profiles (session count, average scroll depth, last seen timestamp) using Prisma ORM, while ClickHouse receives every raw event for high-performance analytical queries. An LLM-powered segmentation endpoint (`POST /segment`) reads user profiles from PostgreSQL and calls OpenAI's GPT-4o-mini API with structured prompt engineering to classify user intent into three categories (browsing, high_intent, ready_to_buy), assign confidence scores, recommend engagement channels (email, SMS, WhatsApp), and generate personalized messages under 100 characters. The system gracefully degrades to rule-based segmentation when the OpenAI API is unavailable. A live analytics dashboard is available at `GET /dashboard` powered by ClickHouse aggregations, and the tracker snippet is served as a static file from the `/public` directory. The entire infrastructure—RabbitMQ, PostgreSQL, and ClickHouse—spins up with a single `docker-compose up -d` command.

> **AI tools used:** OpenAI GPT-4o-mini for behavioral intent classification and personalized message generation. The implementation uses structured prompt engineering with a system message that strictly enforces JSON output format, preventing the LLM from returning markdown or explanatory text. The prompt defines a fixed schema with constrained enums for intent (`browsing` | `high_intent` | `ready_to_buy`) and recommended_channel (`email` | `sms` | `whatsapp`), plus a max 100-character personalized_message field. The service implements graceful degradation: if the OpenAI API key is missing, invalid, or the API call fails, the system automatically falls back to rule-based segmentation using thresholds on session count (≥10 sessions + ≥70% scroll depth = ready_to_buy, ≥4 sessions OR ≥50% scroll depth = high_intent, else browsing). The OpenAI client is lazily initialized at runtime with proper error handling for missing dependencies.

> **What I learned:** Building this pipeline taught me how to architect production-grade event-driven systems that separate high-throughput ingestion from CPU-intensive processing. The RabbitMQ message queue acts as a shock absorber—when traffic spikes, events accumulate in the queue rather than overwhelming the database or causing API timeouts. I learned why columnar storage databases like ClickHouse dramatically outperform row-based systems (PostgreSQL) for analytical workloads: ClickHouse only reads the columns referenced in a query, making aggregations like `SELECT eventType, count() GROUP BY eventType` 10-100x faster on large datasets. I implemented proper message acknowledgment patterns—using `channel.ack()` on successful processing and `channel.nack(msg, false, false)` on failures to prevent poison pills from blocking the queue. The Prisma ORM with PostgreSQL adapter pattern provides type-safe database operations, and I used `upsert()` operations to atomically update existing user profiles or create new ones in a single query, avoiding race conditions. For the browser tracker, I learned to use debouncing (500ms delay) on scroll events to prevent flooding the API, and implemented localStorage-based user ID persistence to track returning visitors across sessions.

> **What I'd improve:** First, I would replace the OpenAI API dependency with a lightweight local transformer model fine-tuned on behavioral sequences—something under 10MB that runs client-side or on-edge, similar to how ATHENA at Markopolo operates. This would eliminate latency, cost, and external API dependency while maintaining privacy. Second, I would implement proper session stitching using fingerprinting techniques beyond localStorage (which gets cleared) to link anonymous visitors across browser sessions and devices. Third, I would add a dead-letter queue (DLQ) for failed message processing rather than just nack-without-requeue, enabling operators to inspect and replay failed events after fixing bugs. Fourth, I would add proper authentication to the `/segment` endpoint (currently unprotected) and implement rate limiting on `/events` to prevent abuse. Finally, I would instrument the codebase with OpenTelemetry tracing to visualize the full event lifecycle from browser → API → RabbitMQ → worker → databases.
